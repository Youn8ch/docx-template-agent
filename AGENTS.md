# AGENTS.md

本文件用于约束 `docx-template-agent` 项目的开发行为。所有代码生成、重构、测试和架构设计都必须遵守本文规则。

## 项目定位

`docx-template-agent` 是一个轻量级内网 docx 模板化排版工具。

第一版只做 **docx 离线处理**，目标是对现有 `.docx` 文档执行受控的格式化操作，并输出新的 `.docx` 文件。

## 参考项目边界

当前仓库中存在参考项目：

```text
ref/wps-agent-master
```

该目录仅作为架构和设计参考：

- 不允许直接修改 `ref/wps-agent-master` 下的任何文件。
- 不允许从参考项目进行全量复制。
- 参考时只分析与 **docx 离线处理** 相关的设计。
- 不分析、不迁移、不实现 Excel、PPT、review、compare、migrate、wps_bridge 等无关模块。
- 如需借鉴参考项目，应提炼设计思想，并用本项目自己的最小实现落地。

## 第一版硬性约束

第一版必须满足以下限制：

- 只支持 docx 离线处理。
- 不使用 WPS COM。
- 不引入 `pywin32`。
- 不依赖 WPS 客户端。
- 不覆盖原始 docx 文件。
- 不修改正文业务内容。
- 不删除段落。
- 不修改表格单元格文本。
- 不一次性引入大而全架构。

## 文档安全原则

处理 `.docx` 时必须遵循保守策略：

- 输入文件必须只读处理。
- 输出文件必须写入新路径，不得覆盖原文件。
- 不得改写正文文本内容。
- 不得删除、合并、重排正文段落。
- 不得修改表格单元格内的文本。
- 格式修改应尽量局限于样式、段落格式、字体格式、页边距、标题层级等明确的排版属性。
- 遇到无法确认是否会影响业务内容的操作，应拒绝执行或降级为只读分析。

## Operations 机制

所有格式修改必须转换为 `operations` 后执行。

开发要求：

- 不允许业务逻辑直接修改 docx。
- 不允许工具层、接口层或模型输出直接操作文档对象。
- 所有可执行修改必须表达为结构化 operation。
- 每个 operation 必须包含明确的类型、目标范围和参数。
- operations 执行前必须经过白名单校验。
- 未进入白名单的 operation 一律拒绝执行。
- operation 校验失败时，应返回清晰错误，不得静默跳过危险操作。

推荐流程：

```text
input docx
  -> analyze
  -> generate operations
  -> validate operations by whitelist
  -> apply operations to copied document
  -> write output docx
```

## 白名单原则

operations 白名单应保持小而明确。

白名单只允许第一版确实需要的格式化能力，例如：

- 设置段落样式。
- 设置标题样式。
- 设置段落对齐方式。
- 设置段前、段后、行距。
- 设置字体名称、字号、加粗等格式属性。
- 设置页面边距等安全页面属性。

禁止进入白名单的能力包括：

- 修改正文文本。
- 替换正文内容。
- 删除段落。
- 删除表格、行、列或单元格。
- 修改表格单元格文本。
- 执行宏、脚本、COM 或客户端自动化。
- 调用 WPS 客户端能力。

## 架构方向

新功能优先保持 `engine` 层独立。

第一版应优先形成清晰的离线处理核心：

- docx 读取与复制。
- 文档结构分析。
- operation 数据结构。
- operation 白名单校验。
- operation 执行器。
- 输出文件生成。
- 基础测试。

后续能力可以在 engine 稳定后再接入：

- MCP。
- opencode Skill。
- 私有化模型。
- Web/API/UI 层。

在这些能力接入前，不应让 engine 依赖它们。

## 依赖约束

依赖选择应保守：

- 不引入 `pywin32`。
- 不引入 WPS COM 相关依赖。
- 不依赖本机安装的 WPS 客户端。
- 优先使用可在服务器或内网环境稳定运行的纯离线库。
- 新增依赖必须有明确用途，避免为未来可能需求提前引入大依赖。

## 测试要求

代码修改后需要尽量补充测试。

测试应优先覆盖：

- operation 白名单校验。
- 禁止修改正文内容。
- 禁止删除段落。
- 禁止修改表格单元格文本。
- 输出文件不覆盖输入文件。
- docx 离线处理流程。
- 对危险 operation 的拒绝行为。

如某次修改无法补充测试，应在交付说明中说明原因和剩余风险。

## 开发方式

开发时应遵循以下原则：

- 小步实现，避免一次性引入大而全架构。
- 优先实现当前需求的最小闭环。
- 保持 engine 与外部接入层解耦。
- 明确区分分析、生成 operations、校验、执行四个阶段。
- 对危险行为默认拒绝。
- 对不确定行为默认保守处理。
- 不为了复用参考项目而牺牲本项目的轻量目标。

## 明确禁止

以下行为在本项目中禁止：

- 修改 `ref/wps-agent-master`。
- 全量复制参考项目。
- 分析或迁移与 docx 离线处理无关的模块。
- 使用 WPS COM。
- 引入 `pywin32`。
- 依赖 WPS 客户端。
- 覆盖原始 docx。
- 修改正文业务内容。
- 删除段落。
- 修改表格单元格文本。
- 绕过 operations 机制直接修改文档。
- 绕过白名单执行 operation。
- 一次性搭建超出当前需求的大型架构。

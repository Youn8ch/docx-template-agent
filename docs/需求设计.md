# docx-template-agent 需求设计

## 1. 项目定位

`docx-template-agent` 是轻量级内网 docx 离线模板化排版工具。

当前版本已经具备 CLI、Web、MCP 和 LLM 辅助分析能力，但主链路仍然是确定性的规则引擎：

```text
输入 docx
  -> 只读解析
  -> 构建 DocumentModel
  -> 按 YAML 模板检查格式
  -> 生成 FormatOperation
  -> 白名单校验
  -> 在复制后的文档上应用格式操作
  -> 输出 formatted.docx、report.md、operations.json
```

LLM 是旁路 advisory 层，只用于辅助判断和解释，不参与最终格式化操作生成，不驱动文档修改。

## 2. 当前能力范围

当前已具备：

- CLI 离线处理。
- Web 上传、检查、格式化和下载。
- MCP 工具调用。
- YAML 模板加载。
- docx 结构解析。
- 标题、正文、表格等基础结构识别。
- 格式问题检查。
- 安全 `FormatOperation` 生成。
- operation 白名单校验。
- 新 docx 输出。
- Markdown 检查报告。
- JSON operations 明细。
- 私有 OpenAI-compatible LLM 辅助分析。
- LLM health check。
- LLM fallback，不阻断主流程。

当前不做：

- WPS COM。
- `pywin32`。
- 依赖 WPS 客户端。
- 覆盖原始 docx。
- 修改正文业务内容。
- 删除段落。
- 修改表格单元格文本。
- 由 LLM 执行排版或驱动格式修改。
- 由 LLM 决定模板并覆盖用户指定模板。

## 3. 主链路与旁路

### 3.1 规则引擎主链路

真正排版依据始终是：

- YAML 模板。
- 规则引擎检查结果。
- `build_operations()` 生成的 `FormatOperation`。
- 白名单校验。
- `apply_operations()` 执行结果。

`operations.json` 只保存规则引擎结果，不保存 LLM 结果。

### 3.2 LLM advisory 旁路

LLM 可用于：

- `document_type`
- `heading_levels`
- `report_summary`
- `template_recommendation`

LLM 不允许：

- 生成或采纳 `FormatOperation`。
- 影响 `build_operations()`。
- 影响 `apply_operations()`。
- 影响 `formatted.docx`。
- 修改正文。
- 修改表格单元格文本。
- 覆盖 CLI `--template` 或调用方指定模板。

LLM 异常时必须 fallback 到规则引擎结果。

## 4. 接入方式

### 4.1 CLI

CLI 支持：

```bash
python app.py llm-test
python app.py --input samples/input/demo.docx --template templates/report.yaml --output samples/output
```

当前 CLI 没有 `--use-llm` 参数。是否启用 LLM 由 `config.yaml` 的 `llm.enabled` 和环境变量决定。

### 4.2 Web

Web 提供：

- 上传 `.docx`。
- 选择模板。
- checkbox：启用 LLM 辅助分析。
- `/check`：执行格式检查。
- `/format`：执行模板排版。

`use_llm=false` 默认不调用 LLM。`use_llm=true` 时输出 `llm_analysis.json`。

### 4.3 MCP

MCP 工具：

- `list_templates`
- `analyze_docx`
- `check_docx_style`
- `apply_docx_template`
- `generate_review_report`

只有 `apply_docx_template` 和 `generate_review_report` 支持 `use_llm`。`list_templates`、`analyze_docx`、`check_docx_style` 不接 LLM。

## 5. LLM 隐私边界

发送给 LLM 的 document snapshot 必须保持最小化：

- 默认不发送全文。
- 顶层包含 `full_text_sent: false`。
- 顶层包含 `paragraph_text_is_preview_only: true`。
- 只发送段落 `text_preview` 和样式摘要。
- `text_preview` 脱敏 IPv4、邮箱、手机号、URL。
- 段落 preview 默认 80 字符。
- 段落数量默认最多 80。
- 表格只发送 `table_count`，不发送单元格内容。

## 6. 安全操作白名单

允许进入执行器的操作必须是明确的格式操作，例如：

- 设置段落样式。
- 设置标题样式。
- 设置段落对齐方式。
- 设置段前、段后、行距。
- 设置首行缩进。
- 设置字体名称、字号、加粗等格式属性。
- 设置安全页面属性。
- 设置表格内段落或 run 的格式属性。

禁止操作包括：

- 修改正文文本。
- 替换正文内容。
- 删除段落。
- 删除表格、行、列或单元格。
- 修改表格单元格文本。
- 合并或拆分正文段落。
- 执行宏、脚本、COM 或客户端自动化。
- 调用 WPS 客户端能力。

未知 operation 默认拒绝。

## 7. 输出产物

一次格式化流程可输出：

- `formatted.docx`：复制输入文档后应用安全格式操作得到的新文档。
- `*_check_report.md`：人工复核报告。
- `*_operations.json`：规则引擎结果。
- `llm_analysis.json`：LLM 辅助分析结果，仅在相关入口启用或 CLI LLM 流程生成时存在。

`llm_analysis.json` 与 `operations.json` 必须分离。

## 8. 验收边界

验收关注：

- 输出 docx 不覆盖输入 docx。
- 正文业务内容保持不变。
- 段落不被删除。
- 表格单元格文本不被修改。
- operations 经白名单校验。
- 危险 operation 被拒绝。
- LLM snapshot 不发送全文。
- LLM 结果不进入 `operations.json`。
- LLM 异常不阻断 `formatted.docx`。
- Web/MCP 默认不调用 LLM。

当前测试验收结果：

```text
63 passed, 1 warning
```

覆盖范围包括 CLI、Web、MCP、LLM fallback、snapshot 脱敏、operations 隔离和 LLM 异常降级。

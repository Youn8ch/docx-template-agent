# docx-template-agent 最小实现方案

## 1. 当前最小闭环

当前实现已经形成 docx 离线模板化排版的最小闭环：

```text
输入 docx
  -> 读取 YAML 模板
  -> 只读解析 docx
  -> 构建 DocumentModel
  -> 识别标题、正文、表格
  -> 按模板检查格式问题
  -> build_operations()
  -> 白名单校验 FormatOperation
  -> apply_operations() 写入新的 formatted.docx
  -> 输出 *_check_report.md
  -> 输出 *_operations.json
  -> 可选输出 llm_analysis.json
```

真正排版依据仍然是 YAML 模板、规则引擎和 `FormatOperation` 白名单。LLM 不自动排版，不自动选择模板，不参与执行操作。

## 3. CLI 实现

CLI 入口为 `app.py`。

支持：

```bash
python app.py llm-test
python app.py --input samples/input/demo.docx --template templates/report.yaml --output samples/output
```

当前 CLI 没有 `--use-llm` 参数。LLM 是否可用由 `config.yaml` 的 `llm.enabled` 和环境变量决定。LLM 不可用或异常时，输出 `rule_only_fallback`。

CLI 主流程会输出：

- `formatted.docx`
- `*_check_report.md`
- `*_operations.json`
- `llm_analysis.json`

## 4. LLM 辅助实现

LLM 相关代码位于：

- `src/llm/private_llm_client.py`
- `src/llm/assistance.py`

`private_llm_client.py` 负责：

- 加载 `config.yaml` 与环境变量。
- 兼容 `endpoint` / `base_url`。
- 调用 OpenAI-compatible `/v1/chat/completions`。
- `health_check()`。
- `max_retries`。
- 构建安全 document snapshot。
- advisory-only 任务：文档类型、标题层级、报告摘要、模板推荐。

`assistance.py` 负责：

- 统一构建 LLM assistance。
- 捕获 LLM 异常。
- 生成 `rule_only_fallback`。
- 写出 `llm_analysis.json`。

LLM snapshot 边界：

- `full_text_sent: false`
- `paragraph_text_is_preview_only: true`
- 只发送 `text_preview` 和样式摘要。
- `text_preview` 脱敏 IPv4、邮箱、手机号、URL。
- preview 默认 80 字符。
- 段落默认最多 80。
- 表格只发送 `table_count`。

## 5. Web use_llm

Web 入口位于：

- `src/web/app.py`
- `src/web/service.py`

页面提供 “启用 LLM 辅助分析” checkbox。

行为：

- `/check` 支持 `use_llm`。
- `/format` 支持 `use_llm`。
- `use_llm=false` 默认不调用 LLM。
- `use_llm=true` 生成 `llm_analysis.json`。
- `/format` 中 LLM 异常不影响 `formatted.docx`。
- `operations.json` 不包含 LLM 结果。

## 6. MCP use_llm

MCP 入口位于：

- `mcp_server.py`
- `src/mcp/tools.py`

工具：

- `list_templates`
- `analyze_docx`
- `check_docx_style`
- `apply_docx_template`
- `generate_review_report`

LLM 边界：

- 只有 `apply_docx_template` 和 `generate_review_report` 支持 `use_llm`。
- `list_templates`、`analyze_docx`、`check_docx_style` 不接 LLM。
- `use_llm=false` 默认不调用 LLM。
- `use_llm=true` 输出 `llm_analysis.json`。
- `apply_docx_template` 中 LLM 异常不影响 `formatted.docx`。
- `operations.json` 不包含 LLM 结果。

## 7. 输出文件

格式化流程输出：

```text
*_formatted.docx
*_check_report.md
*_operations.json
llm_analysis.json
```

说明：

- `*_formatted.docx` 是复制输入文档后应用安全格式操作得到的新文档。
- `*_check_report.md` 面向人工复核，可包含 “LLM 辅助分析” 章节。
- `*_operations.json` 是规则引擎结果，只包含 issues、operations、validation、results。
- `llm_analysis.json` 是 LLM 辅助分析结果，仅供参考。

## 8. fallback 策略

LLM 不得阻断主流程。

当 LLM 未启用、配置不完整、连接失败、返回异常或解析异常时：

- 返回 `rule_only_fallback`。
- `operations_source` 保持 `rule_engine_only`。
- `operations_generated` 保持 `false`。
- 继续使用 YAML 模板和规则引擎结果。
- 已进入格式化流程时继续输出 `formatted.docx`。

## 9. 安全边界

禁止：

- 覆盖输入 docx。
- 修改正文业务内容。
- 删除段落。
- 修改表格单元格文本。
- 让 LLM 生成或采纳 `FormatOperation`。
- 让 LLM 影响 `build_operations()`。
- 让 LLM 影响 `apply_operations()`。
- 让 LLM 推荐模板覆盖用户指定模板。
- 把 LLM 返回结果写入 `operations.json`。

允许：

- 修改白名单内的格式属性。
- 输出 LLM advisory 分析到独立的 `llm_analysis.json`。
- 在 Markdown 报告中展示 LLM 辅助摘要。
